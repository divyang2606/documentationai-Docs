## Pricing

SproutOS is completely free. There are no tiers, no paywalls, and no features locked behind a paid plan. Both the GitHub version and the WordPress.org version are free. The only difference is that the GitHub version includes PHP execution, which is disabled on WordPress.org to comply with directory guidelines.

Novamira has a genuinely useful free tier: PHP execution, filesystem tools, and sandbox management - 10 tools in total. Pro starts at €49/year for 3 sites and adds Elementor abilities, Bricks abilities, WordPress content management (posts and pages), and cross-session memory.

Respira requires a paid subscription from the start. Maker covers one site at €5.92/month (billed €71/year). Builder covers 5 sites at €12.50/month. Studio covers 25 sites at €25.75/month. WooCommerce is a separate add-on at +€95/year. An accessibility add-on costs €49/month on top of that.

WP-CLI is free, but it is not an AI tool. See the [WP-CLI section](#wp-cli-not-the-same-thing) below.

## Page Builder Support

SproutOS ships dedicated ability modules for Elementor and Bricks. Both are free and activate automatically when the builder is installed. On top of that, SproutOS includes a generic cross-builder layer with 18 tools - add, move, remove, reorder, patch, and batch-update elements without builder-specific knowledge.

Respira supports the most builders: Gutenberg, Divi 4 and 5, Elementor, Oxygen (Classic and Oxygen 6), Bricks, WPBakery, Beaver Builder, Thrive Architect, Brizy, Visual Composer, and Breakdance - 11 total. Each builder has its own schema and intelligence layer that helps AI understand the data format. This breadth is a real advantage if your agency works across many builders.

Novamira's Pro tier adds Elementor and Bricks. Other builders are not covered natively, though PHP execution lets AI work with any builder indirectly through code.

## PHP Execution and Code Automation

SproutOS and Novamira both support PHP execution. Respira does not.

With SproutOS, you can ask your AI to write a PHP snippet, test it against your live WordPress environment, iterate on errors, and deploy working code to the sandbox - all in one session. This opens a class of tasks that named tools cannot cover on their own: custom reports, one-off database operations, plugin interactions, performance diagnostics, and anything that needs running code to produce a result.

<Callout kind="info">
PHP execution in SproutOS requires the GitHub build. The WordPress.org build disables it to comply with directory guidelines. Both builds are free.
</Callout>

Respira's 234 named tools and builder-specific endpoints cover a wide surface area. But when a task falls outside what a named tool handles, there is no code execution fallback. SproutOS uses PHP execution as that fallback - your AI can always write and run code when the right tool does not exist.

## Safety Approach

All three plugins take a different approach to safety. None is strictly safer - they make different trade-offs suited to different workflows.

**SproutOS** uses a layered model. Safe Mode makes every ability read-only with one toggle. The Sandbox isolates all AI-written PHP into a dedicated directory so code never touches core files directly. Crash Guard detects fatal errors, auto-disables the offending file, and activates Safe Mode automatically until you review it. Meta snapshots capture state before destructive changes. Protected files (`.env`, `wp-config.php`, `.htaccess`) and protected directories (`wp-admin/`, `wp-includes/`, plugin and theme folders) are hard-blocked at the code level.

**Novamira** uses a sandbox for AI-written PHP with the same crash recovery pattern - a broken file gets disabled and safe mode activates. There is no Safe Mode toggle or per-tool control.

**Respira** takes a duplicate-first approach. AI edits a copy of your page by default, not the original. You review and approve the change before it goes live. Every edit captures a before/after snapshot with a version timeline, visual preview, and one-click restore. This is the most conservative of the three and suits client sites where you want a human approval step before anything publishes.

## Token Efficiency

Every tool definition your AI receives at session start consumes tokens from your context window. This adds up across long sessions or multi-site workflows.

SproutOS is the only plugin here with Capability Profiles - a system that controls how many abilities are sent to your AI client upfront. The default Minimal profile sends roughly 1,500 tokens (~6 KB). The Ultra-Minimal profile sends only the three Bridge tools and lets AI discover everything else on demand.

| Profile | Upfront cost | Best for |
|---|---|---|
| Ultra-Minimal | Bridge tools only | Token-sensitive sessions |
| Minimal (default) | ~1,500 tokens / ~6 KB | Most workflows |
| Standard | Moderate catalogue | Builder-heavy sessions |
| Full | Complete catalogue | Maximum tool depth |

Novamira sends approximately 14 KB upfront. Respira sends approximately 25 KB. Neither has a way to reduce this.

Bridge tools are available in all profiles including Ultra-Minimal. Your AI can still reach any of the 175+ abilities on demand through `sprout-bridge/dispatch-tool` - the token savings do not limit what is possible.

## Analytics and Monitoring

SproutOS is the only plugin here with a built-in analytics dashboard. Every tool call is logged with a timestamp, the WordPress user who triggered it, the ability name, the risk level (readonly, additive, destructive, or unknown), and execution time in milliseconds. The dashboard shows six stat cards: total calls, calls today, failed calls (highlighted in red), unique sessions, unique users, and most-used ability.

You can configure webhook notifications to Slack, Discord, or any custom endpoint. Filter events by all ability calls, destructive-only, or errors-only. Payloads are HMAC-signed with an `X-Sprout-Signature` header. There is also a CSV export and a one-click log purge.

Respira has an audit log that records all API activity - a useful record, but no dashboard or outbound notifications. Novamira has no logging or monitoring.

## Per-Tool Control

SproutOS lets you enable or disable any of the 175+ abilities individually from the AI Abilities tab. You can allow file reads but block file writes. You can keep Elementor enabled but turn off PHP execution. Every toggle takes effect immediately without restarting anything.

Respira has tool governance - you can apply a per-site allowlist in whitelist mode, blacklist mode, or full-access mode. This is broader control at a higher level of granularity.

Novamira has no per-tool controls. All tools are on or all tools are off.

## Architecture

SproutOS and Novamira are both pure WordPress plugins. Install, activate, and connect - no additional software required on your local machine beyond your AI client.

Respira requires a Node.js MCP server installed separately on your local machine. You download it from respira.press, configure it with your API key and site URL, and run it alongside your AI client. This adds a setup step and an ongoing dependency that needs to stay running. The upside is that the Node server adds capabilities at the transport layer - per-call watchdogs, upload timeouts, multi-site switching, and agent telemetry headers.

## WP-CLI - Not the Same Thing

WP-CLI comes up in almost every conversation about AI and WordPress, but it is a different category of tool entirely.

WP-CLI is a command-line interface for WordPress management. It runs commands like `wp post list`, `wp plugin activate`, or `wp db query`. It does not understand natural language - you write the exact command, flags, and syntax. There is no AI layer and no MCP endpoint.

SproutOS with an AI client lets you say "check all plugins for security issues and tell me which ones are risky to update." WP-CLI requires you to already know: `wp plugin list --update=available --format=table`.

Many developers use both. WP-CLI is fast for quick manual tasks when you know exactly what to run. SproutOS is for workflows where you want AI to figure out what to do, execute it, and report back - without you needing to know the underlying commands.
