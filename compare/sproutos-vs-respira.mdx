SproutOS is a free WordPress-only plugin with PHP execution, Capability Profiles, and a built-in analytics dashboard. Respira is a paid plugin that also requires a separate Node.js server running on your local machine. Respira covers 11 page builders and uses a human-approval workflow before changes go live. SproutOS covers Elementor and Bricks and uses an automated safety model. Both are designed for agencies managing multiple WordPress sites.

## Pricing

SproutOS is free. No subscription, no paid tier, no features locked behind payment. The GitHub build and WordPress.org build are both free. PHP execution is only available on the GitHub build due to WordPress.org directory guidelines.

Respira requires a paid plan from day one:

| Plan | Price | Sites |
|---|---|---|
| Maker | €5.92/mo (€71/yr) | 1 |
| Builder | €12.50/mo (€150/yr) | 5 |

| Accessibility add-on | +€49/mo | All plans |

An agency on 5 sites with WooCommerce pays at least €245/year on Respira. The same setup on SproutOS costs €0.

## Architecture

SproutOS is a WordPress-only plugin. Install it, activate it, add your credentials to your AI client. No other software required. It connects via HTTP for remote sites or via STDIO for local environments.

Respira requires two components:

1. A WordPress plugin installed on your site
2. A Node.js server installed separately on your local machine

The Node server runs locally and proxies requests to your WordPress REST API. It needs to stay running throughout every session. If it crashes or you switch machines, the connection drops.

The Node layer adds real capabilities: per-call watchdogs, upload timeouts, multi-site switching, and agent telemetry headers. These are not possible from a plugin alone.

## Page Builder Support

Respira supports 11 page builders with dedicated schema layers for each: Gutenberg, Divi 4, Divi 5, Elementor, Oxygen Classic, Oxygen 6, Bricks, WPBakery, Beaver Builder, Thrive Architect, Brizy, Visual Composer, and Breakdance. Each integration gives your AI structured knowledge of how that builder stores page data. For agencies that work across many different site stacks, this is a significant advantage.

SproutOS includes dedicated ability modules for Elementor and Bricks, both free, plus 18 cross-builder abilities for Elementor, Bricks, and Gutenberg. For unsupported builders like WPBakery or Beaver Builder, PHP execution lets your AI interact through code rather than named tools.

## PHP Execution

SproutOS supports PHP execution. Your AI can write a snippet, run it in the live WordPress environment, read the output, and iterate until the task is done. This covers custom reports, one-off database operations, plugin interactions, and anything that falls outside the named tool catalogue.

Respira does not support PHP execution. When a task falls outside its 234 named tools, there is no code-based fallback.

<Callout kind="info">
PHP execution in SproutOS requires the GitHub build. Both the GitHub and WordPress.org builds are free.
</Callout>

## Safety Models

Respira uses a duplicate-first, human-approval model. Your AI edits a copy of the page, not the original. You review the change in a visual preview and approve it before it goes live. Every change is version-tracked with a one-click restore. This is the most conservative approach available and suits workflows where client sites need human sign-off before anything publishes.

SproutOS uses an automated safety model:

- **Safe Mode toggle:** one click makes every ability read-only
- **The Sandbox:** AI-written PHP runs in `wp-content/sproutos-mcp-sandbox/`, isolated from core files
- **Crash Guard:** a fatal PHP error disables the broken file and activates Safe Mode automatically
- **Protected files:** `.env`, `wp-config.php`, `.htaccess` are hard-blocked from all AI access
- **Protected directories:** `wp-admin/`, `wp-includes/`, plugin and theme folders cannot be deleted
- **Per-tool control:** disable any ability individually without affecting others

Respira is more human-in-the-loop. SproutOS is more automated. Neither is strictly safer — they suit different workflows.

## Token Usage

SproutOS has 4 Capability Profiles that control how many abilities are sent to your AI client at session start:

| Profile | Upfront cost | Best for |
|---|---|---|
| Ultra-Minimal | Bridge tools only | Token-sensitive sessions |
| Minimal (default) | ~1,500 tokens / ~6 KB | Most workflows |
| Standard | Moderate catalogue | Builder-heavy sessions |
| Full | Complete catalogue | Maximum tool depth |

All 175+ abilities remain reachable in every profile via `sprout-bridge/dispatch-tool`.

Respira sends approximately 25 KB upfront with no way to reduce it. On models with a limited effective context window, this is a meaningful constraint in long sessions.

---

## Analytics and Monitoring

SproutOS logs every ability call with a timestamp, WordPress user, ability name, risk level, and execution time. The dashboard shows six stat cards. Webhook notifications push events to Slack, Discord, or any custom endpoint. Payloads are HMAC-signed. CSV export and log purge are included.

Respira has an audit log that records all API activity. There is no dashboard, no outbound notifications, and no risk-level tagging per call.

## WooCommerce

SproutOS includes WooCommerce abilities free: products, orders, categories, tags, and sales reports. The module activates automatically when WooCommerce is active.

Respira charges +€95/year for WooCommerce on top of your base plan. On the Maker plan, adding WooCommerce brings the annual cost to €166/year for one site.

## Frequently Asked Questions

<Expandable title="Can SproutOS and Respira run on the same WordPress site?">
  Both install as WordPress plugins with separate endpoints. Running both at once on the same site is not a tested configuration. There is no known technical conflict, but managing two overlapping tool sets from one AI session creates confusion.
</Expandable>

<Expandable title="Does Respira's higher tool count make it more capable?">
  Not across the board. Respira's 234 tools reflect 11 builder integrations, many of which are builder-specific. SproutOS's PHP execution covers tasks that no fixed tool list can handle. For Elementor and Bricks, the tool count difference is less meaningful. For Divi, Oxygen, or Beaver Builder, Respira has dedicated tools that SproutOS does not.
</Expandable>

<Expandable title="How difficult is Respira's Node.js server to set up?">
  It adds at least one extra step compared to a plugin-only install. You need Node.js on your machine, the server package from respira.press, your API key, and your site URL configured. For developers, this is straightforward. For non-developers, the ongoing dependency is a higher bar.
</Expandable>

<Expandable title="What does SproutOS do when no named tool covers the task?">
  PHP execution is the fallback. Your AI writes a PHP snippet, runs it via `sprout/execute-php`, inspects the output, and iterates. This covers most tasks outside the 175+ named abilities. Requires the GitHub build.
</Expandable>

<Expandable title="Does SproutOS support Divi or Beaver Builder?">
  Not with dedicated modules. SproutOS covers Elementor, Bricks, and Gutenberg. For Divi, WPBakery, Beaver Builder, Oxygen, and others, PHP execution lets your AI work through code rather than a named integration.
</Expandable>
